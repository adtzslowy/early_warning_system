🔄 Data Flow: Real-time Monitoring

1️⃣ Data Acquisition (Poll dari IoT API)
Trigger: Artisan command `rob:sync` (polling berkala, default 5 menit)
Flow:
RobSyncService::sync()
→ Fetch dari endpoint `$config['services.iot.base_url']`
→ Parse response JSON ke DTO (RobDevice)
→ Upsert Device record, store Sensor history
Output: DB terisi device status + sensor readings (raw: jarak ultrasonik, suhu, kelembapan, dll)
2️⃣ Data Transformation & Feature Engineering
Service yang terlibat:
OnshoreWindCalculator, RiseRateCalculator, LunarPhaseCalculator
Proses:
  • Jarak ultrasonik → Tinggi muka air (invers)
  • Arah + kecepatan angin → Komponen onshore (proyeksi kosinus)
  • Pembacaan 2 jam lalu ↔ sekarang → Laju kenaikan air
  • Tanggal/waktu → Fase bulan (untuk faktor spring tide)
Output: Feature set yang siap untuk Fuzzy + Prediction
3️⃣ Risk Classification (Fuzzy Mamdani)
Input: 3 variabel (ketinggian air, laju kenaikan, komponen angin onshore)
Process:
FuzzyEngine → Fuzzification → Rule Inference → Defuzzification
Output: Skor risiko 0-100 → kategori (Aman/Waspada/Siaga/Bahaya)
Storage: RiskEvaluation record disimpan per device
4️⃣ Water Level Prediction (Linear Regression)
Input: Historical data (ketinggian air, angin, gelombang, fase bulan) dari 24 jam terakhir
Process:
PredictionFeatureBuilder → LinearRegression::fit() → predict() untuk 8 horizons (15-120 min)
Model retraining: Setiap 15 menit menggunakan OLS (Ordinary Least Squares)
Output: Prediction record untuk setiap horizon waktu
5️⃣ Display di Dashboard (Real-time via AJAX Polling)
Endpoint:
GET /dashboard/devices/{device}/snapshot (JSON)
Data yang dikembalikan: Status terkini (level, angin, suhu, risiko, prediksi)
Update frequency: AJAX polling setiap 30-60 detik (configurable)

🧬 Core Components

📡 RobSyncService

Orchestrator utama: fetch data IoT, simpan ke DB, trigger events

HTTP GET ke IoT endpoint
DTO parsing (RobDevice)
DB transaction (atomicity)
Event dispatch (DeviceStatusChanged, SensorDataSaved)

🌊 SyntheticRobDataGenerator

Generate mock data untuk testing & demo (tanpa sensor fisik)

Pasang surut diurnal (sinus 24 jam)
Surge (gelombang + angin onshore)
Model diurnal (suhu, kelembapan, tekanan)
Jitter deterministik (reproducible noise)

💨 OnshoreWindCalculator

Ekstrak komponen angin yang relevan untuk wind setup

Input: kecepatan + arah angin
Output: komponen onshore (m/s)
Bearing pantai Ketapang: 270° (Barat)

📈 RiseRateCalculator

Hitung laju kenaikan muka air (early warning indicator)

Lookback: 60 menit
Formula: (d_lama - d_baru) / Δt
Output: cm/jam

🌙 LunarPhaseCalculator

Hitung fase bulan → faktor pasang purnama (spring tide risk)

Perioda: 29.53 hari sinodis
Normalisasi ke [0,1]
1.0 = spring tide (risiko max)

🧠 Fuzzy Mamdani Engine

Klasifikasi tingkat risiko multi-parameter

3 input: ketinggian, laju, angin
4 output: Aman, Waspada, Siaga, Bahaya
Method: Max-Min (Min implikasi, Max komposisi)
Defuzzifikasi: Centroid (CoG)

📊 WaterLevelPredictor

Prediksi ketinggian air multi-horizon (8 horizons: 15-120 min)

Algorithm: Multiple Linear Regression (MLR)
Features: 5 prediktor (level, laju, angin, fase bulan)
Retraining: Setiap 15 menit
Solver: OLS (Gauss-Jordan dengan partial pivoting)

🎨 DashboardController

Orchestrate data untuk real-time visualization

Snapshot endpoint: JSON response
Detail endpoint: historical data
Eager load relations (N+1 prevention)
Permission gating per device

🗄️ Database Schema (Key Models)

Model Key Fields Purpose
Device device_code, name, latitude, longitude, status, last_seen_at Master sensor (monitoring point)
Sensor device_id, type, value, unit, recorded_at Raw readings history (time-series)
RiskEvaluation device_id, risk_score, risk_level, evaluated_at Fuzzy output (risk classification)
Prediction device_id, horizon_minutes, predicted_level, predicted_at Linear regression forecast (8 horizons)
Alert device_id, risk_level, triggered_at, acknowledged_at Alert history (untuk notification system)
User, Role, Permission Multi-level access control (RBAC) Authentication & authorization

⚡ Background Jobs & Scheduling

Command: `rob:sync` (Poll IoT API)
Trigger: Setiap 5 menit via Laravel scheduler
Action: RobSyncService::sync() → upsert devices, store sensors, dispatch events
Command: `RefreshBmkgMaritime` (Update weather forecast)
Trigger: Setiap jam (0 * * * *)
Action: Fetch prakiraan angin & gelombang dari BMKG API untuk kalibrasi model
Command: `DetectOfflineDevices` (Check device heartbeat)
Trigger: Setiap 10 menit
Action: Tandai device offline jika tidak ada pembacaan > 15 menit
Command: `BacktestPrediction` (Validation & accuracy check)
Trigger: Manual (untuk testing & tuning)
Action: Jalankan prediksi ulang terhadap historical data, bandingkan dengan aktual

👤 User Interactions & Workflows

1️⃣ Authentication & Authorization
Sign In: Email + Password OR Google OAuth
Email Verification: Required (grace period: show dashboard sambil pending verification)
RBAC: Roles (Admin, Operator, Viewer) + granular permissions per action
Device Visibility: Users hanya bisa lihat devices sesuai permission mereka (visibleTo scope)
2️⃣ Dashboard Monitoring (Main Interface)
GET /dashboard
Display:
  • Device list (table) dengan status terkini + worst-risk highlighting
  • Device detail panel (dipilih user atau auto = highest risk device)
  • Real-time metrics: ketinggian air, suhu, angin, risiko, prediksi
  • Charts: historical telemetry (24h) + prediction curve (8h ke depan)
  • Map: Leaflet.js dengan device markers (status visual)
Real-time updates: AJAX polling ke `/dashboard/devices/{device}/snapshot` setiap 30-60 detik
3️⃣ Device Management (Master Data CRUD)
GET /devices — List all devices (paginated)
POST /devices — Create new device (name, latitude, longitude, area_code BMKG)
PATCH /devices/{device} — Update device info
DELETE /devices/{device} — Remove device
Permission guard: Semua aksi memerlukan `permission:view devices`
4️⃣ Sensor Data Browsing
GET /sensor — List sensor readings (dengan filter device)
GET /sensor/{sensor} — Show detail pembacaan + derived features (wind component, rise rate)
Use case: Operator mau verify raw data, check kalibrasi sensor, investigate anomalies
5️⃣ Alert Management
GET /alert — View alert history (trigger time, risk level, device, acknowledged status)
Data source: Alert model (diisi saat risk_level naik ke Siaga/Bahaya via event listener)
Future enhancement: Real-time notification (email/SMS/push) saat alert triggered
6️⃣ User & Role Management (Admin Only)
GET /users — List users + roles
POST /users — Create user (email, role, permissions)
PATCH /users/{user} — Update user role/permissions
DELETE /users/{user} — Deactivate user
Permission guard: `permission:view users`, `permission:create users`, dll
7️⃣ Settings & Preferences (Per User)
GET /settings — View display preferences (theme, timezone, units)
PUT /settings — Update preferences (stored in DisplayPreferences DTO)
GET /settings/profile — View user profile (email, name)
PATCH /settings/profile — Update profile info

💡 Key Design Decisions

Why Polling (not WebSocket)?
Simpler infrastructure, stateless server, no persistent connections. Polling interval (30-60s) adequate untuk early warning use case.
Why Synthetic Data?
Allows comprehensive testing tanpa sensor fisik. Model parameter diturunkan dari referensi lokal (Ketapang). Jitter deterministik memastikan reproducibility.
Why Fuzzy + Linear Regression (dual approach)?
Fuzzy: Real-time risk classification (multi-faktor, intuitive rules). Linear Regression: Forward-looking prediction (8 horizons) untuk early warning lead time.
Why Separate Services for Calculations?
Single Responsibility: OnshoreWindCalculator, RiseRateCalculator, LunarPhaseCalculator terpisah, testable, reusable.
Why AJAX Polling over WebSocket?
Dashboard tidak butuh real-time push (IoT data masuk setiap 5 min). Polling 30-60s cukup. Avoid complexity of WebSocket servers, connection management, fallback handling.
Why Display Preferences DTO?
Decouple user preferences dari database structure. Timezone-aware display, unit conversion (cm vs inch), theme preference — semua client-side rendered.
